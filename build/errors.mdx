## HTTP status codes

The API uses standard HTTP status codes:

| Code | Meaning |
|------|---------|
| `200` | Success |
| `400` | Bad request (invalid parameters or body) |
| `401` | Unauthorized (missing or invalid API key) |
| `403` | Forbidden (no permission for this resource) |
| `404` | Not found |
| `422` | Unprocessable entity (validation error) |
| `429` | Too many requests (rate limited) |
| `500` | Server error |

## Error response format

Error responses typically include a JSON body with details (e.g. message, code, or validation errors). Check the [API Reference](/api-reference/introduction) for schemas. When in doubt, use the `message` field for a human-readable description.

## Rate limits

If rate limits are enforced, you will receive `429 Too Many Requests`. Retry after the period indicated in the response (e.g. `Retry-After` header or message). Implement backoff and respect any documented rate limits to avoid throttling.

## Authentication errors

- **401** – Ensure the `Authorization: Bearer YOUR_API_KEY` header is set and the key is valid.
- **403** – Your key may not have permission for the requested resource or operation.
